# Proxy LXC: Caddy + Cloudflare Tunnel + Cloudflare MCP

This guide documents the **current preferred proxy pattern** for this homelab:

> Run **Caddy**, **Cloudflare Tunnel (`cloudflared`)**, and **Cloudflare MCP** together in **one dedicated proxy LXC**.

Do **not** create separate Caddy and Cloudflared LXCs for the normal rebuild unless you intentionally want to split them later. The older separate-LXC approach is more confusing and not needed for the current setup.

## LXC requirements

When recreating the proxy LXC, keep the current homelab LXC defaults:

```text

unprivileged: 0
```

In Proxmox UI this means:

- Enable nesting.
- `Unprivileged container = No`.
- Privileged container.
- Keep CPU/memory close to current unless intentionally resizing:

```text

swap: 512
cpulimit: 1
rootfs: 6G on vm_storage
```

Record final CT ID/IP in:

```text

```

## Secrets policy

Do **not** commit real Cloudflare tokens/API keys/account IDs.

Use placeholders in repo examples and store real values only on the proxy LXC, password manager, or sealed backup.

Secrets used by this LXC:

| Secret | Used by | Where to create/get it | Minimum useful scope |
| --- | --- | --- | --- |
| `CF_API_TOKEN` / `CLOUDFLARE_API_TOKEN` | Caddy DNS challenge, Caddy DDNS, Cloudflare MCP | Cloudflare Dashboard -> Profile -> API Tokens -> Create Token | `Zone:DNS:Edit`, `Zone:Zone:Read` for the specific zone |
| Cloudflare Tunnel token | `cloudflared.service` | Cloudflare Zero Trust -> Networks -> Tunnels -> your tunnel -> Install and run connector | Tunnel connector token generated by Cloudflare |
| Cloudflare Account ID | Cloudflare MCP | Cloudflare Dashboard URL/account overview or Zero Trust account settings | Not a password, but still keep out of public examples |

## Recommended rebuild order

### 1. Create one proxy LXC

Recommended simple path:

1. Use the Community Scripts **Caddy** LXC as the base.
2. During Advanced setup, set the LXC to this homelab's defaults:

```text
features: nesting=1
unprivileged: 0
hostname: proxy
ip: 192.168.1.201/24
```

Community Scripts page:

```text
https://community-scripts.github.io/ProxmoxVE/scripts?id=caddy
```

Raw PVE shell command pattern:

```bash
bash -c "$(curl -fsSL https://raw.githubusercontent.com/community-scripts/ProxmoxVE/main/ct/caddy.sh)"
```

Important: this creates the **proxy LXC**. Do not also run the Cloudflared community script as a second LXC for the standard rebuild.

### 2. Install or verify Caddy Cloudflare modules

Caddy needs the Cloudflare DNS provider module. The current live config also uses the dynamic DNS module.

Expected modules:

```text
dns.providers.cloudflare
dynamic_dns
```

Check:

```bash
caddy list-modules | grep -E 'dns.providers.cloudflare|dynamic_dns'
```

If missing, rebuild Caddy with both modules:

```bash
apt-get update
apt-get install -y curl git build-essential

# Install Go if it is not already available.
# Use Debian package if good enough, or install Go manually if xcaddy requires newer Go.
apt-get install -y golang-go

go install github.com/caddyserver/xcaddy/cmd/xcaddy@latest
/root/go/bin/xcaddy build \
  --with github.com/mholt/caddy-dynamicdns \
  --with github.com/caddy-dns/cloudflare

systemctl stop caddy
cp "$(command -v caddy)" /usr/bin/caddy.bak.$(date +%Y%m%d-%H%M%S)
install -m 0755 ./caddy /usr/bin/caddy
systemctl start caddy
caddy list-modules | grep -E 'dns.providers.cloudflare|dynamic_dns'
```

### 3. Store Cloudflare API token for Caddy

Prefer a systemd override instead of shell profile exports:

```bash
systemctl edit caddy
```

Add:

```ini
[Service]
Environment=CF_API_TOKEN=replace-with-real-cloudflare-token
Environment=CLOUDFLARE_API_TOKEN=replace-with-real-cloudflare-token
```

Then reload and restart:

```bash
systemctl daemon-reload
systemctl restart caddy
systemctl status caddy --no-pager
```

Why both names? Current tools commonly use either `CF_API_TOKEN` or `CLOUDFLARE_API_TOKEN`. Caddyfile examples use `{env.CF_API_TOKEN}`. Cloudflare MCP commonly expects `CLOUDFLARE_API_TOKEN`.

### 4. Restore Caddyfile

Sanitized reusable template:

```text
config/Caddyfile.example
```

Current live pattern:

- Global Caddy admin listens on `0.0.0.0:2019`.
- Caddy dynamic DNS updates Cloudflare DNS for selected names.
- Wildcard site uses Cloudflare DNS challenge.
- Backends reverse proxy to LAN service IPs/ports.

Copy and edit:

```bash
cp config/Caddyfile.example /tmp/Caddyfile
nano /tmp/Caddyfile
caddy fmt --overwrite /tmp/Caddyfile
caddy validate --config /tmp/Caddyfile
cp /tmp/Caddyfile /etc/caddy/Caddyfile
systemctl restart caddy
systemctl status caddy --no-pager
```

Security note: `admin 0.0.0.0:2019` exposes Caddy's admin API on the LAN. Keep it only if protected by firewall/LAN rules. Safer default is `admin localhost:2019` or disabling remote admin.

### 5. Install Cloudflare Tunnel in the same LXC

Install `cloudflared` in the **same proxy LXC**.

Example Debian install pattern:

```bash
mkdir -p --mode=0755 /usr/share/keyrings
curl -fsSL https://pkg.cloudflare.com/cloudflare-main.gpg \
  | tee /usr/share/keyrings/cloudflare-main.gpg >/dev/null

echo 'deb [signed-by=/usr/share/keyrings/cloudflare-main.gpg] https://pkg.cloudflare.com/cloudflared bookworm main' \
  | tee /etc/apt/sources.list.d/cloudflared.list

apt-get update
apt-get install -y cloudflared
```

If the package installs to `/usr/bin/cloudflared` but you want to match the current live layout, either leave it there or add a symlink:

```bash
ln -sf "$(command -v cloudflared)" /usr/local/bin/cloudflared
```

### 6. Create Cloudflare Tunnel and service

Cloudflare path:

```text
Cloudflare Dashboard -> Zero Trust -> Networks -> Tunnels
```

Create/select the tunnel, then copy the connector token from:

```text
Your tunnel -> Install and run connector
```

Install as a service:

```bash
cloudflared service install '<cloudflare-tunnel-token-from-dashboard>'
systemctl enable --now cloudflared
systemctl status cloudflared --no-pager
```

The current live service shape is:

```ini
[Service]
ExecStart=/usr/bin/cloudflared --no-autoupdate tunnel run --token [REDACTED]
Restart=on-failure
RestartSec=5s
```

The live setup does **not** currently rely on `/etc/cloudflared/config.yml`; it runs from the tunnel token in systemd.

### 7. Add public hostnames in Cloudflare Tunnel

In the Cloudflare Tunnel UI, add public hostnames for each service:

1. Enter the service subdomain.
2. Choose the owned domain.
3. Enter the internal LAN URL and port.
4. Type:
   - `HTTP`: simplest for normal LAN services.
   - `HTTPS`: if the service enables HTTPS by default, open `Additional settings -> TLS` and enable `No TLS Verify` for self-signed/internal certs.

For sensitive public services, add Cloudflare Access login:

```text
Zero Trust -> Access -> Applications -> Add an application -> Self-hosted
```

## Cloudflare MCP in the proxy LXC

Current live setup has a wrapper script:

```text
/usr/local/bin/cloudflare-mcp
```

It runs the official Cloudflare MCP server through `npx`:

```bash
exec npx --yes @cloudflare/mcp-server-cloudflare run "${CLOUDFLARE_ACCOUNT_ID}"
```

Recommended sanitized wrapper:

```bash
#!/usr/bin/env bash
set -euo pipefail

export PATH=/usr/local/bin:/opt/node-v22.15.0-linux-x64/bin:$PATH

# Keep this value out of Git.
export CLOUDFLARE_ACCOUNT_ID="replace-with-cloudflare-account-id"

# Prefer a systemd/user environment variable. Fallback can reuse Caddy's token
# if you intentionally store it in Caddy's systemd override.
if [ -z "${CLOUDFLARE_API_TOKEN:-}" ]; then
  token="$(systemctl show caddy --property=Environment --value \
    | tr ' ' '\n' \
    | sed -n 's/^CF_API_TOKEN=//p' \
    | head -n1)"
  if [ -n "$token" ]; then
    export CLOUDFLARE_API_TOKEN="$token"
  fi
fi

exec npx --yes @cloudflare/mcp-server-cloudflare run "${CLOUDFLARE_ACCOUNT_ID}"
```

Install/update wrapper:

```bash
nano /usr/local/bin/cloudflare-mcp
chmod 0755 /usr/local/bin/cloudflare-mcp
```

Verify manually without printing secrets:

```bash
CLOUDFLARE_API_TOKEN='replace-with-token' \
CLOUDFLARE_ACCOUNT_ID='replace-with-account-id' \
cloudflare-mcp
```

If it starts, stop it with `Ctrl+C`. It is an MCP stdio server, so it is normally launched by an agent/client, not as a long-running HTTP service.

## Verification checklist

Run in the proxy LXC:

```bash
systemctl status caddy --no-pager
caddy validate --config /etc/caddy/Caddyfile
caddy list-modules | grep -E 'dns.providers.cloudflare|dynamic_dns'

systemctl status cloudflared --no-pager
cloudflared --version

node --version
npm --version
/usr/local/bin/cloudflare-mcp --help || true
```

Then test from LAN and outside LAN:

- Public Cloudflare Tunnel hostname resolves.
- Cloudflare Access login appears where expected.
- Caddy wildcard hosts obtain certificates.
- Internal reverse proxies reach the correct LXC IP/port.

## Troubleshooting

### `cloudflared tunnel list` complains about missing origin cert

This is expected when the connector is installed from a tunnel token and not authenticated with a local origin certificate:

```text
Cannot determine default origin certificate path
```

Use the Cloudflare Dashboard tunnel page or `systemctl status cloudflared` for connector health instead.

### Caddy cannot solve DNS challenge

Check:

```bash
systemctl show caddy --property=Environment
caddy list-modules | grep dns.providers.cloudflare
journalctl -u caddy -n 100 --no-pager
```

Common causes:

- Missing `CF_API_TOKEN` in the Caddy systemd environment.
- Token scoped to the wrong Cloudflare zone.
- Caddy binary does not include `dns.providers.cloudflare`.

### Cloudflare MCP starts but agent cannot use it

Check:

```bash
command -v node npm npx
command -v cloudflare-mcp
cloudflare-mcp
```

Common causes:

- Missing `CLOUDFLARE_ACCOUNT_ID`.
- Missing `CLOUDFLARE_API_TOKEN`.
- Agent config points to the wrong command/path.
